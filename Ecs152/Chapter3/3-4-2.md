## Principles of Reliable Data Transfer (Part 2)

We now want to extend our reliable data transfer protocol to cover the case where not only can the channel corrupt messages, but the channel can also lose messages. This applies to both the data flowing from sender to receiver, as well as acknowledgements flowing from the receiver back to the sender.

The rdt3.0 protocol handles the case of lost packets.

---

### rdt3.0: Channels with Errors and Loss

New channel assumption:
- underlying channel can also lose packets (data, ACKs)

Approach:
- sender waits a "reasonable" amount of time for an ACK
- if packet (or ACK) is lost:
  - sender retransmits

If a packet (or ACK) is just delayed (not lost):
- retransmission will be a duplicate, but sequence numbers already handle this
- receiver must specify the sequence number of the packet being ACKed

## Performance of rdt3.0 (stop-and-wait)

Usender: utilization — fraction of time sender is busy sending

Example:
- 1 Gbps link
- 15 ms propagation delay
- 8000 bit packet

Time to transmit packet into channel:

Dtrans = L / R = 8000 bits / (10^9 bits/sec) = 8 microseconds

## Go-Back-N: sender

Sender:
- "window" of up to N consecutive transmitted but unACKed packets
- k-bit sequence number in packet header

<img width="561" height="123" alt="Screenshot 2026-04-17 at 2 16 25 PM" src="https://github.com/user-attachments/assets/ac5fa9d8-824b-403c-aacc-92c3ee55fcf5" />

Cumulative ACK:
- ACK(n) acknowledges all packets up to and including sequence number n

On receiving ACK(n):
- move window forward to begin at n + 1

Timer:
- timer for oldest in-flight packet

On timeout(n):
- retransmit packet n and all higher sequence number packets in the window

---

## Go-Back-N: Receiver

Whenever it receives a packet, the receiver sends an acknowledgement for the correctly received packets so far with the highest in-order sequence number.

ACK-only:
- always send ACK for the highest in-order correctly received packet
- may generate duplicate ACKs
- only need to remember rcv_base

On receipt of out-of-order packet:
- can discard (do not buffer) or buffer (implementation decision)
- re-ACK packet with highest in-order sequence number

## Selective Repeat

Receiver:
- individually acknowledges all correctly received packets
- buffers packets as needed for eventual in-order delivery to upper layer

Sender:
- times out and retransmits individually for unACKed packets
- maintains a timer for each unACKed packet

## Summary

rdt protocol mechanisms:
- error detection (e.g., checksum)
- ACKs, NAKs
- retransmission
- sequence numbers (duplicate detection)
- timeout and retransmission (lost messages)
